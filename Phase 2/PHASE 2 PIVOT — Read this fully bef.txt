--- CHANGE 1: STORAGE LAYER ---

Replace Scaleway S3 with local disk. This is the only change to the API and daemon.

In packages/api/:
- Replace S3Client and all presigned URL logic with a LocalStorageService class
- LocalStorageService reads files from LOCAL_MEDIA_PATH using Node.js fs streams
- GET /stream/:id streams the file directly from disk with full HTTP Range
  request support (Accept-Ranges: bytes) so seeking works in the video player
- GET /stream/:id/subtitle reads the subtitle file from disk and returns it
- Upload endpoints (used by the ingest daemon's registrar) accept a local
  file path instead of an S3 key
- The s3_key column in the DB is repurposed to store a relative local path
  (e.g. anime/01JTXYZ/s1/e1/episode.mp4) — no schema migration needed,
  just a semantic change
- Remove all boto3/S3 environment variables from config

In ingest/:
- Remove uploader.py entirely (no S3 upload step)
- Replace it with mover.py: after FFmpeg transcodes the .mp4, move it to
  LOCAL_MEDIA_PATH/<s3_key_convention_path> using the /mnt/d/ WSL2 path
- Update registrar.py to pass the local relative path instead of an S3 key
- Update the FFmpeg command in transcoder.py to use h264_nvenc as described above
- Everything else in the daemon — RSS polling, Nyaa HTML scraping, qBittorrent
  integration, filename parsing, backlog mode, sequential batch downloading,
  subtitle extraction, Jikan metadata registration — is completely untouched

--- ZERO-TOUCH OPERATION REQUIREMENT ---

The Alienware must be completely hands-off once set up. I will never physically
interact with it after initial setup. It just sits connected to the router via
Ethernet and runs. Every change I make to the codebase on my development machine
must automatically sync to and restart on the Alienware without me touching it.

This requirement affects how everything is deployed and updated:

1. AUTO-SYNC VIA GIT + PM2
   Set up a bare git remote on the Alienware (in WSL2) so that when I do
   git push alienware main from my dev machine, a post-receive git hook
   automatically:
   - Pulls the latest code into /mnt/d/Babylon/app/
   - Runs pnpm install if package.json changed
   - Runs pnpm build for packages/api and packages/web
   - Runs pm2 reload babylon-api and pm2 reload babylon-web
   This means a single git push from my laptop deploys everything.
   No SSH required after initial setup.

2. API AUTO-RESTART
   PM2 must be configured with pm2 startup so it:
   - Starts automatically when Windows boots (via PM2's Windows startup script)
   - Automatically restarts babylon-api and babylon-web if they crash
   - Saves the process list (pm2 save) so it survives reboots

3. INGEST DAEMON AUTO-RESTART (WSL2)
   The Python ingest daemon runs in WSL2. Configure it to:
   - Start automatically when WSL2 starts using a WSL2 startup mechanism
     (add the daemon launch to /etc/wsl.conf [boot] section or a systemd
     service within WSL2 if systemd is enabled)
   - Restart automatically on crash (use a simple supervisor loop in the
     daemon itself, or WSL2 systemd service with Restart=always)
   - When git push alienware main fires the post-receive hook, the hook
     must also restart the ingest daemon:
     sudo systemctl restart babylon-ingest
     (or equivalent depending on how WSL2 auto-start is implemented)

4. QBITTORRENT AUTO-START
   qBittorrent (Windows native) must be configured to:
   - Launch on Windows startup via the Windows startup folder or Task Scheduler
   - Always start minimized to system tray, never require interaction

5. WINDOWS AUTO-LOGIN + NEVER SLEEP
   For all of the above to work across reboots the Alienware must:
   - Auto-login to Windows on boot without requiring a password entry
     (configure via netplwiz)
   - Never sleep or hibernate: set power plan to High Performance,
     disable all sleep/hibernate/screen timeout in Power Options,
     disable USB selective suspend
   - Close lid without sleeping: set lid close action to Do Nothing
     in Power Options (so the laptop can sit closed)

   Include these Windows configuration steps explicitly in the
   setup guide so I can do them once during initial setup.

6. NETWORK STATIC IP
   Assign a static LAN IP to the Alienware in the router's DHCP reservation
   table using the Ethernet MAC address. Include instructions in the setup
   guide for how to find the MAC address (ipconfig /all) and where to set
   the reservation (generic instructions since router admin UIs vary).
   Once set, the Alienware is always reachable at the same IP on the LAN
   regardless of reboots.

The setup guide at deploy/PHASE2_LOCAL_SETUP.md must include every one of
these steps in order, written as a one-time setup checklist. After following
it once, I should never need to touch the Alienware again.
